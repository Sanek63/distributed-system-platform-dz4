# Лабораторная работа: Saga и брокер сообщений

Цель работы: реализовать систему обработки заказов на основе паттерна Saga (оркестрация) и асинхронного взаимодействия через брокер сообщений — и убедиться в корректности обработки отказов и компенсаций.
- Необходимо реализовать брокер сообщений, Saga-оркестратор, три сервиса-участника (склад, оплата, доставка) и клиент, отправляющий заказы.

#### Базовая модель

- Все компоненты — независимые Docker-контейнеры: брокер, оркестратор, три сервиса-участника и клиент.
- Межсервисное взаимодействие идёт **только через брокер сообщений**. Клиент общается с оркестратором напрямую по HTTP.
- Брокер сообщений — HTTP-сервис, реализующий хранение и доставку сообщений по топикам с гарантией at-least-once.
- Оркестратор управляет жизненным циклом заказа: последовательно отправляет команды сервисам через брокер и при отказе запускает компенсирующие действия в обратном порядке.
- Каждый сервис-участник подписывается на свой топик команд, обрабатывает команды (основные и компенсирующие) и публикует результат в топик результатов.
- Сервисы обязаны быть **идемпотентными**: повторная обработка одного и того же сообщения не должна дублировать эффект.

Saga — последовательность из трёх шагов:
1. **T1: Резерв товара** (Warehouse) → **C1: Отмена резерва**
2. **T2: Списание оплаты** (Payment) → **C2: Возврат оплаты**
3. **T3: Назначение доставки** (Delivery) → **C3: Отмена доставки**

При отказе на шаге Tᵢ запускаются компенсации Cᵢ₋₁, …, C₁ в обратном порядке.

#### Брокер сообщений

Брокер — центральный компонент для асинхронной коммуникации. Конфигурируется переменными окружения: `BROKER_PORT`, `REDELIVERY_TIMEOUT_MS` (таймаут до повторной доставки неподтверждённого сообщения).

Модель данных:
- **Topic** — именованный упорядоченный журнал сообщений (append-only log).
- **Subscriber** — именованный подписчик на топик; каждый подписчик имеет собственный указатель чтения (offset). Разные подписчики одного топика получают все сообщения независимо друг от друга.
- Сообщение считается доставленным только после явного подтверждения (ack). Если ack не получен за `REDELIVERY_TIMEOUT_MS` — сообщение доставляется повторно при следующем запросе `consume`.

API:

- `POST /topics` — создать топик.
    - Тело запроса:
        ```json
        { "name": <string> }
        ```
    - Ответ при успехе: `201 Created`
    - Ответ если топик уже существует: `409 Conflict`

- `POST /subscribe` — подписать потребителя на топик.
    - Тело запроса:
        ```json
        { "topic": <string>, "subscriber": <string> }
        ```
    - Ответ при успехе: `200 OK`
    - Ответ если топик не найден: `404 Not Found`

- `POST /publish` — опубликовать сообщение в топик.
    - Тело запроса:
        ```json
        {
            "topic": <string>,
            "key": <string>,
            "value": <string>
        }
        ```
    - Ответ при успехе: `200 OK`
        ```json
        { "id": <string>, "offset": <int> }
        ```
    - Ответ если топик не найден: `404 Not Found`
    - `key` используется для логической группировки сообщений (например, `orderId`); `value` — произвольная строка (JSON-encoded полезная нагрузка). Для брокера `key` и `value` — непрозрачные данные.

- `GET /consume?topic=<string>&subscriber=<string>` — получить следующее необработанное сообщение для данного подписчика.
    - Ответ при наличии сообщения: `200 OK`
        ```json
        {
            "id": <string>,
            "offset": <int>,
            "key": <string>,
            "value": <string>,
            "timestamp": <int>
        }
        ```
    - Ответ если нет новых сообщений: `204 No Content`
    - Если подписчик ранее получил сообщение, но не подтвердил его, и прошло `REDELIVERY_TIMEOUT_MS` — то же сообщение доставляется повторно.

- `POST /ack` — подтвердить обработку сообщения.
    - Тело запроса:
        ```json
        { "topic": <string>, "subscriber": <string>, "id": <string> }
        ```
    - Ответ при успехе: `200 OK`

- `GET /status` — состояние брокера.
    - Ответ: `200 OK`
        ```json
        {
            "topics": [
                {
                    "name": <string>,
                    "messageCount": <int>,
                    "subscribers": [
                        {
                            "name": <string>,
                            "processedOffset": <int>,
                            "pendingMessages": <int>
                        }
                    ]
                }
            ]
        }
        ```

#### Оркестратор (Saga)

Оркестратор — управляющий компонент, реализующий Saga-оркестрацию. Конфигурируется переменными окружения: `ORCHESTRATOR_PORT`, `BROKER_URL`.

При запуске оркестратор:
1. Создаёт на брокере топики: `warehouse-commands`, `payment-commands`, `delivery-commands`, `warehouse-results`, `payment-results`, `delivery-results`.
2. Подписывается (subscriber = `orchestrator`) на топики результатов: `warehouse-results`, `payment-results`, `delivery-results`.
3. Запускает фоновый цикл опроса (polling) топиков результатов.

Внешний API (для клиента):

- `POST /order` — создать новый заказ.
    - Тело запроса:
        ```json
        {
            "items": [
                { "item": <string>, "quantity": <int> }
            ],
            "paymentAmount": <int>,
            "deliveryAddress": <string>
        }
        ```
    - Ответ при успехе: `200 OK`
        ```json
        { "orderId": <string>, "status": "pending" }
        ```

- `GET /order/<orderId>` — получить статус заказа.
    - Ответ при наличии заказа: `200 OK`
        ```json
        {
            "orderId": <string>,
            "status": <"pending"|"warehouse-reserving"|"payment-processing"|"delivery-scheduling"|"compensating"|"completed"|"cancelled">,
            "completedSteps": [<string>],
            "compensatedSteps": [<string>]
        }
        ```
    - Ответ при отсутствии заказа: `404 Not Found`

- `GET /orders` — получить список всех заказов.
    - Ответ: `200 OK`
        ```json
        {
            "orders": [
                { "orderId": <string>, "status": <string> }
            ]
        }
        ```

Логика Saga (оркестрация):

**Прямой путь (forward):**
1. При получении нового заказа: `status = "pending"`, публикует в `warehouse-commands`:
    ```json
    { "orderId": "<id>", "action": "reserve", "items": [{"item": "<string>", "quantity": <int>}] }
    ```
    `status = "warehouse-reserving"`
2. При получении успешного результата из `warehouse-results`: публикует в `payment-commands`:
    ```json
    { "orderId": "<id>", "action": "charge", "amount": <int> }
    ```
    `status = "payment-processing"`
3. При получении успешного результата из `payment-results`: публикует в `delivery-commands`:
    ```json
    { "orderId": "<id>", "action": "schedule", "address": "<string>", "items": [{"item": "<string>", "quantity": <int>}] }
    ```
    `status = "delivery-scheduling"`
4. При получении успешного результата из `delivery-results`:
    `status = "completed"`

**Обратный путь (compensation):**
При получении неуспешного результата на любом шаге оркестратор переходит в `status = "compensating"` и последовательно запускает компенсации для всех уже выполненных шагов в обратном порядке:
- Если T3 (delivery) упал → компенсировать: C2 (refund), затем C1 (cancel-reserve).
- Если T2 (payment) упал → компенсировать: C1 (cancel-reserve).
- Если T1 (warehouse) упал → компенсировать нечего.

Компенсирующие команды:
- `cancel-reserve` в `warehouse-commands`: `{ "orderId": "<id>", "action": "cancel-reserve" }`
- `refund` в `payment-commands`: `{ "orderId": "<id>", "action": "refund" }`
- `cancel-delivery` в `delivery-commands`: `{ "orderId": "<id>", "action": "cancel-delivery" }`

Оркестратор ожидает успешного результата от каждой компенсации прежде чем запускать следующую. После завершения всех компенсаций: `status = "cancelled"`.

#### Сервисы-участники

Каждый сервис — отдельный Docker-контейнер. Общие переменные окружения: `SERVICE_PORT`, `BROKER_URL`. При запуске каждый сервис подписывается на свой топик команд и в цикле опрашивает брокер.

**Идемпотентность:**
Каждый сервис хранит множество обработанных `messageId`. При получении сообщения с уже обработанным `messageId` сервис не выполняет действие повторно, а только заново публикует ранее сохранённый результат и отправляет ack.

**Формат результата** (публикуется в `{service}-results`, поле `value` — JSON-строка):
```json
{
    "orderId": <string>,
    "action": <string>,
    "status": "success" | "failure",
    "message": <string>
}
```

Поле `key` при публикации результата совпадает с `orderId`.

---

**Warehouse Service (склад)**

Хранит в памяти остатки товаров и выполненные резервы. Конфигурируется: `INITIAL_INVENTORY` (начальные остатки, формат: `item1:100,item2:50`).

- Подписывается на `warehouse-commands` (subscriber = `warehouse`).
- При получении `action: "reserve"`:
    - Проверяет наличие всех запрошенных товаров в нужном количестве.
    - Если товар есть — уменьшает остатки, сохраняет резерв по `orderId`, публикует `status: "success"` в `warehouse-results`.
    - Если товара недостаточно — публикует `status: "failure"`.
- При получении `action: "cancel-reserve"`:
    - Возвращает зарезервированные товары на склад по `orderId`, удаляет резерв, публикует `status: "success"` в `warehouse-results`.

API для мониторинга:
- `GET /status`
    ```json
    {
        "inventory": { "<item>": <int> },
        "reservations": { "<orderId>": [{"item": "<string>", "quantity": <int>}] },
        "processedMessages": <int>
    }
    ```

---

**Payment Service (оплата)**

Хранит в памяти баланс и выполненные списания. Конфигурируется: `INITIAL_BALANCE` (начальный баланс, целое число).

- Подписывается на `payment-commands` (subscriber = `payment`).
- При получении `action: "charge"`:
    - Проверяет достаточность баланса.
    - Если баланс достаточен — списывает сумму, сохраняет списание по `orderId`, публикует `status: "success"` в `payment-results`.
    - Если баланс недостаточен — публикует `status: "failure"`.
- При получении `action: "refund"`:
    - Возвращает ранее списанную сумму на баланс по `orderId`, удаляет списание, публикует `status: "success"` в `payment-results`.

API для мониторинга:
- `GET /status`
    ```json
    {
        "balance": <int>,
        "charges": { "<orderId>": <int> },
        "processedMessages": <int>
    }
    ```

---

**Delivery Service (доставка)**

Хранит в памяти список назначенных доставок. Конфигурируется: `FAIL_ACTION` (опционально; если задано значение `schedule`, сервис возвращает `failure` на все команды `schedule` — используется для тестовых сценариев).

- Подписывается на `delivery-commands` (subscriber = `delivery`).
- При получении `action: "schedule"`:
    - Если `FAIL_ACTION=schedule` — публикует `status: "failure"` в `delivery-results`.
    - Иначе — создаёт запись о доставке по `orderId`, публикует `status: "success"` в `delivery-results`.
- При получении `action: "cancel-delivery"`:
    - Удаляет запись о доставке по `orderId`, публикует `status: "success"` в `delivery-results`.

API для мониторинга:
- `GET /status`
    ```json
    {
        "deliveries": { "<orderId>": { "address": "<string>", "items": [{"item": "<string>", "quantity": <int>}] } },
        "processedMessages": <int>
    }
    ```

#### Клиент

Клиент — отдельный процесс или скрипт, работающий в непрерывном цикле. Логика одинакова для всех сценариев:

1. Отправляет `POST /order` на оркестратор с данными заказа.
2. Периодически опрашивает `GET /order/<orderId>` до достижения финального статуса (`completed` или `cancelled`).
3. Проверяет согласованность состояния сервисов через `GET /status` на каждом сервисе:
    - Если `completed`: товар зарезервирован на складе, оплата списана, доставка назначена.
    - Если `cancelled`: все промежуточные эффекты компенсированы — товар возвращён, оплата возвращена, доставка отменена.
    - Если состояние сервисов не соответствует финальному статусу заказа — фиксирует несогласованность.
4. Обновляет счётчики:
    - `completed` — успешно завершённые заказы.
    - `cancelled` — корректно отменённые заказы (все компенсации выполнены, состояние согласовано).
    - `inconsistent` — заказ в финальном статусе, но состояние сервисов не согласовано.
    - `errors` — сетевые ошибки или тайм-ауты.

Формат вывода в терминал (обновляется на месте каждую итерацию):
```
completed: 5 | cancelled: 2 | inconsistent: 0 | errors: 0
```

#### Сценарий 1. Успешная обработка заказа

- Все сервисы запущены и работают нормально. `INITIAL_INVENTORY=item1:100,item2:50`, `INITIAL_BALANCE=10000`.
- Запустить клиент, отправить заказ с `items: [{item: "item1", quantity: 2}]`, `paymentAmount: 500`.
- Ожидаемое поведение:
    - Оркестратор последовательно выполняет T1 → T2 → T3, каждый шаг завершается успешно.
    - Статус заказа: `completed`.
    - `GET /status` на складе: `inventory.item1 = 98`, в `reservations` есть запись для данного заказа.
    - `GET /status` на оплате: `balance = 9500`, в `charges` есть запись для данного заказа.
    - `GET /status` на доставке: в `deliveries` есть запись для данного заказа.
    - Лог: `completed` растёт, `inconsistent` и `errors` остаются нулевыми.

#### Сценарий 2. Недостаточный баланс — компенсация резерва

- `INITIAL_INVENTORY=item1:100`, `INITIAL_BALANCE=50`.
- Отправить заказ с `paymentAmount: 500`.
- Ожидаемое поведение:
    - T1 (reserve) выполняется успешно: товар зарезервирован.
    - T2 (charge) завершается с `failure`: баланс недостаточен.
    - Оркестратор переходит в `compensating` и запускает C1 (cancel-reserve).
    - Статус заказа: `cancelled`.
    - `GET /status` на складе: товар возвращён в `inventory`, запись из `reservations` удалена.
    - `GET /status` на оплате: баланс не изменился (`balance = 50`), записей в `charges` нет.
    - Лог: `cancelled` растёт, `inconsistent` остаётся нулевым.

#### Сценарий 3. Отказ доставки — компенсация оплаты и резерва

- Все сервисы запущены. `INITIAL_INVENTORY=item1:100`, `INITIAL_BALANCE=10000`. Delivery Service запущен с `FAIL_ACTION=schedule`.
- Отправить заказ.
- Ожидаемое поведение:
    - T1 (reserve) и T2 (charge) выполняются успешно.
    - T3 (schedule) завершается с `failure`.
    - Оркестратор переходит в `compensating` и последовательно запускает: C2 (refund) → C1 (cancel-reserve).
    - Статус заказа: `cancelled`.
    - `GET /status` на оплате: баланс восстановлен.
    - `GET /status` на складе: товар возвращён.
    - Это демонстрация обратного пути Saga: все выполненные шаги компенсируются в обратном порядке.

#### Сценарий 4. Повторная доставка сообщения после таймаута

- Все сервисы запущены и работают нормально. `REDELIVERY_TIMEOUT_MS=3000`.
- Отправить заказ.
- Во время обработки перезапустить один из сервисов (например, Warehouse) после получения сообщения, но до отправки ack.
- Ожидаемое поведение:
    - Брокер не получает ack в течение `REDELIVERY_TIMEOUT_MS`.
    - Брокер повторно доставляет сообщение (`redeliveryCount` увеличивается).
    - После перезапуска сервис получает повторное сообщение и обрабатывает его.
    - Заказ успешно завершается.
    - Это демонстрация гарантии at-least-once на уровне брокера: сообщение не теряется при сбое потребителя.

#### Сценарий 5. Несколько параллельных заказов

- Все сервисы запущены. `INITIAL_INVENTORY=item1:100`, `INITIAL_BALANCE=10000`.
- Клиент отправляет 10 заказов параллельно (каждый с `paymentAmount: 500`, `items: [{item: "item1", quantity: 5}]`).
- Ожидаемое поведение:
    - Все 10 заказов обрабатываются: каждый проходит полный цикл Saga.
    - Финальное состояние согласовано: сумма резервов на складе + текущий остаток = начальный остаток; сумма списаний + текущий баланс = начальный баланс.
    - Оркестратор корректно ведёт состояние каждого заказа независимо, не путая шаги разных Saga.
    - Лог: `completed` = 10, `inconsistent` = 0.
